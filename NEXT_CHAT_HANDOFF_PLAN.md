# AI 解題網站：整合、題型與提示詞分層計畫

> 交接用途：此文件整理目前已確認的方向與後續執行順序。  
> 狀態：**僅規劃，尚未依本文件移動檔案或重構程式。**

## 1. 專案與目標

- 專案位置：`C:\Users\User\Desktop\AI解題(7.1格式正常)`
- 主網站：`index.html`
- 使用情境：使用者上傳化學題目圖片或輸入文字，由 AI 產生繁中詳解。

本輪後續工作有四個目標：

1. 將目前分散的檔案、資料夾與維護工具整理成容易理解、容易維護的結構。
2. 將提示詞改為分層模組，避免不同規則互相衝突。
3. 在主解題頁新增可複選的「題型」設定，讓 AI 使用對應的解題規則。
4. 將舊資料庫與既有詳解轉為可查詢的參考資料，而不是直接覆蓋 AI 的答案。

## 2. 已確認的使用者需求

### 2.1 主提示詞與分層規則

- 需要一份**主提示詞**，但要寬鬆：負責繁中、教學風格、大致詳細程度，不要塞入過多硬性規定。
- 未來可以增加「詳細版」主提示詞；目前先不做。
- 其他規則要分層管理，不能互相搶控制權。
- 「化學反應方程式表達」、「計算精簡」、「參考答案」等功能，不能只依靠提示詞；網頁必須有可選的設定與後續檢查。

### 2.2 題型複選

- 在**主解題頁**新增題型複選，不必先建立另一個獨立頁面。
- 使用者可選 1～3 個題型；不選時讓 AI 自行判斷。
- 建議初始題型：
  - 化學計量
  - 氧化還原
  - 酸鹼中和
  - 化學平衡
  - 反應速率
  - 電化學
  - 溶液／溶解度
  - 熱化學
  - 有機化學
  - 原子結構／化學鍵
- 範例：高錳酸根滴定題可同時選 `氧化還原`、`化學計量`、`酸鹼中和`。

### 2.3 排版與 NOTE

- 排版功能濃縮為**一個排版工具頁**：處理 LaTeX、分式、數字樣式、中文間距、預覽與輸出。
- NOTE 功能獨立為**另一個工具頁／功能區**：處理註記規則、內容檢查、單位與重點提示。
- 主網頁可自動使用兩者的核心功能；管理介面則分開，避免工具頁太複雜。

### 2.4 舊 database 與詳解

- 不把整個舊 `database/` 或完整舊詳解直接塞進主提示詞。
- 將可用資料拆成：方法規則、格式範例、參考答案、待整理舊資料。
- 每題僅取少量相關資料供 AI 參考。
- 參考答案只用來核對；不可以不經檢查就直接覆蓋 AI 結論。

## 3. 現況摘要

### 3.1 主要區域

| 類別 | 現有內容 | 用途 |
|---|---|---|
| 主網站 | `index.html`、`js/`、`css/`、`prompts/` | 上傳題目、呼叫 AI、顯示詳解 |
| 教師頁面 | `teacher-tools.html`、`knowledge-studio.html`、`method-library.html`、`problem-analyzer.html`、`answer-audit.html`、`evaluation-lab.html` | 教師與資料管理工具 |
| 排版／化學工具 | `solution-format.html`、`chemistry-workbench.html`、`molfile-preview.html` | 排版、化學式與結構工具 |
| 資料 | `knowledge/`、`structures/`、`database/` | 知識、結構式、既有題庫／格式資料 |
| 維護腳本 | `sync-*.py`、`rebuild-*.py`、`fetch-*.py`、`convert-*.py`、`.bat` | 同步、轉檔、重建、啟動等工作 |
| 測試 | `run-self-test.py`、`test-*.html`、`verify-build.html`、`tools/check-js-syntax.py` | 功能與排版基本檢查 |
| 系統資料 | `.git/`、`.agents/`、`.cursor/` | 版本紀錄與工具設定，不應任意移動 |

### 3.2 目前 AI 回覆流程

```text
題目圖片／文字
  → 主網頁組合題目與解題規則
  → AI 讀題並產生詳解（主要 API 呼叫）
  → 視品質檢查結果，必要時可能有一次短暫的 AI 修正
  → 本機 LaTeX 整理、KaTeX 繪製、詳解排版
  → 顯示給使用者
```

- 本機 LaTeX 整理與畫面排版不會額外消耗 API Token。
- 「排版」頁目前可手動使用；尚未成為完整的可視化中繼驗收站。
- 舊 database 的自動解題比對目前已停用；但部分舊檔仍可能被主網站載入，因此**不可直接搬走**。

### 3.3 已知排版問題

- 部分純文字數字，例如「速率變為 4 倍」，與中文距離仍太近。
- 原因：目前公式（KaTeX）與純文字數字經過不同處理；純文字數字尚未統一包裝成可套用樣式的元素。
- 後續需要在排版核心建立一致的 `board-number`／數字格式規則；不能只調公式的 CSS。
- 部分不完整 LaTeX 會裸露顯示，例如 `\dfrac`、`C_{...}`；未來排版工具需要攔截、驗證、標示問題，而不是靜默顯示。

## 4. 目標資料夾結構

```text
AI解題/
├─ index.html                 # 主解題網站入口，保留於根目錄
├─ app/                       # 主網站程式、樣式、提示詞模組
├─ pages/                     # 教師工具、排版工具、NOTE 工具等頁面
├─ data/                      # 知識庫、結構式、題庫與格式資料
├─ scripts/                   # 同步、重建、轉檔、維護工具
├─ tests/                     # 自動檢查與排版測試
├─ docs/                      # 中文使用說明、專案地圖、維護紀錄
└─ legacy/                    # 已確認退休、但暫時保留的舊功能／資料
```

> 搬遷原則：先建立新位置與相容引用，再更新連結，再測試；不可先搬檔再補救。  
> `index.html` 保留在根目錄，避免啟動方式與既有連結失效。

## 5. 提示詞與規格的分層設計

```text
第 1 層：主提示詞（風格與總原則）
第 2 層：網頁選項規格（使用者明確勾選的要求）
第 3 層：題型規則（依複選題型載入）
第 4 層：題目相關知識庫／方法資料
第 5 層：參考答案核對規則
第 6 層：局部修正規則（僅針對檢查到的問題）
```

### 各層唯一責任

| 層級 | 能做的事 | 不應做的事 |
|---|---|---|
| 主提示詞 | 語言、教學語氣、概略詳略 | 不指定每類題的固定步驟 |
| 網頁選項 | 指定要反應式、精簡計算、參考答案等 | 不提供題目知識內容 |
| 題型規則 | 指定該題型的重要檢查點與輸出需求 | 不改變整體語氣 |
| 知識庫 | 提供方法、事實、短範例 | 不覆蓋使用者選項 |
| 參考答案 | 比對最終答案、提出疑點 | 不直接強改答案 |
| 局部修正 | 修正被檢查到的單一缺漏 | 不重新生成整份詳解 |

## 6. 題型選擇與「真正強制」流程

題型選擇不只是加一句提示詞；應產生一份結構化的解題規格：

```text
使用者勾選題型／選項
  → 建立 solveSpec（本機結構化規格）
  → 組合相對應的題型提示詞與資料
  → AI 產生詳解
  → 本機檢查必要項目是否存在
  → 缺漏時僅請 AI 補修該部分，或標示給使用者
```

### 功能範例

| 使用者設定 | AI 的要求 | 網站的檢查 |
|---|---|---|
| 氧化還原 | 說明氧化數／電子轉移；需要時寫半反應式 | 是否包含必要氧化還原步驟 |
| 化學計量 | 優先處理莫耳數、係數比、限量試劑與單位 | 是否有關鍵比例或單位結論 |
| 酸鹼中和 | 說明中和關係、當量點或 pH 判斷 | 是否有酸鹼關鍵判斷 |
| 要求化學反應方程式 | 列平衡反應式；需要時列反應表 | 是否真的有反應式／反應表 |
| 計算精簡 | 保留必要算式與結論，避免長篇重複文字 | 是否超出可接受的冗長格式 |
| 參考答案 | 以參考答案做核對 | 結論是否一致；不一致時標示並複核 |

## 7. 資料庫與既有詳解的整合方式

### 7.1 建議資料分層

```text
data/
├─ knowledge/                 # 概念、方法、教師規則
├─ structures/                # .mol 化學結構式與索引
├─ reference/
│  ├─ methods/                # 已驗證的解題方法
│  ├─ formats/                # 詳解格式範例
│  ├─ answers/                # 參考答案／答案鍵
│  └─ legacy-import/          # 從舊 database 轉入、待人工確認的資料
└─ database-legacy/           # 僅在確認完全不被主網站依賴後再使用
```

### 7.2 使用原則

- 題目分類後，只檢索少量相關規則或範例。
- 不將完整舊詳解、整個題庫一次傳給 AI。
- 每筆參考資料要有題型標籤、來源、可信度／人工確認狀態。
- 若沒有高信心相關資料，寧可不提供，不要錯誤套用。
- 主網頁可顯示「已套用：氧化還原方法、計量格式規則」等透明狀態。

## 8. 排版與 NOTE 的目標分工

### 排版工具頁

- 統一處理 LaTeX 分隔符、合法性檢查、公式渲染、分式留白、數字字型、中文與數字間距。
- 提供原文、修正後預覽、問題清單、可複製／輸出內容。
- 需要建立「未渲染 LaTeX 偵測」，例如 `\dfrac`、`C_{...}` 出現在普通文字時必須標示。

### NOTE 工具頁

- 管理 NOTE 規則、題目註記、單位檢查、重點提示與內容一致性檢查。
- 不負責一般字型與公式外觀。

### 主網頁

- 自動使用兩者的核心檢查。
- 不顯示複雜的管理面板；僅顯示必要狀態與可理解的錯誤提示。

## 9. 分階段執行計畫

### Phase 0：安全基準

- 建立目前檔案用途清單。
- 固定一組排版驗收案例：一般文字、純文字數字、LaTeX 分式、化學式、選項、NOTE。
- 執行既有自動測試，記錄基準結果。

**影響：無功能改動。**

### Phase 1：文件與分類地圖

- 建立 `docs/PROJECT_MAP.md`、`docs/MAINTENANCE_GUIDE.md`、`docs/FILE_STATUS.md`。
- 為每個主要檔案／資料夾標記：使用中、維護工具、測試、待封存。
- 不移動網站運行檔案。

**影響：無功能與排版改動。**

### Phase 2：低風險資料夾整合

- 先整理文件、測試、維護腳本。
- 將測試工具集中到 `tests/`，維護腳本集中到 `scripts/`。
- 每個腳本增加中文用途說明與安全提醒。
- 保留必要的根目錄啟動捷徑或相容入口。

**驗收：主網站、所有工具頁、既有測試皆可使用。**

### Phase 3：提示詞模組化

- 建立主提示詞、網頁選項、題型規則、知識庫資料、參考答案、局部修正的模組邊界。
- 建立「提示詞優先順序與不可覆蓋規則」文件。
- 先維持目前輸出效果；不立刻增加詳細版主提示詞。

**驗收：原有題目結果可產生；提示詞不重複、不矛盾。**

### Phase 4：題型複選與解題規格

- 在主網頁建立題型複選 UI（1～3 項）。
- 建立 `solveSpec`，讓選項不再只是零散文字拼接。
- 為初始題型建立最小可用的規則與檢查器。
- 顯示「已啟用題型／規則」狀態。

**驗收：選取題型後，AI 輸出可觀察到對應步驟；未選取時保持原行為。**

### Phase 5：選項強制檢查與參考答案流程

- 將「反應方程式」、「計算精簡」、「參考答案」改為：網頁設定 + AI 要求 + 本機驗證。
- 對明確缺漏採局部修正；不重做整份詳解。
- 參考答案不一致時，只標示與複核，不強制竄改。

**驗收：可確認設定是否真的被遵守；額外 AI 呼叫只發生在明確缺漏時。**

### Phase 6：舊 database 與詳解資料遷移

- 建立新資料格式與題型標籤。
- 用轉換工具逐步匯入舊資料，保留來源與確認狀態。
- 建立本機檢索，只取相關的少量方法／範例。
- 在完全脫離舊 database 前，保留相容層。

**驗收：資料可被正確選用；沒有資料時不影響一般解題。**

### Phase 7：排版核心與 NOTE 分離

- 排版工具統一數字、文字、LaTeX 與公式的顯示流程。
- NOTE 工具獨立管理內容規則。
- 修正純文字數字與中文間距、數字字型不一致、裸露 LaTeX 等問題。

**驗收：固定排版案例可通過視覺比較；主網頁與工具頁結果一致。**

### Phase 8：正式搬遷與 legacy 封存

- 只有在相容層與測試都通過後，才搬移仍在使用的程式。
- 已確認未使用的舊檔，移至 `legacy/` 並附封存說明。
- 先保留一段觀察期，再考慮刪除。

**驗收：所有入口、連結、資料路徑、測試與排版均正常。**

## 10. 建議執行順序與核准方式

建議順序：

```text
Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5 → Phase 6 → Phase 7 → Phase 8
```

可單獨核准的指令範例：

- 「執行 Phase 1，只建立文件地圖。」
- 「執行 Phase 3，先整理提示詞分層，不改主網頁。」
- 「執行 Phase 4，但先只做氧化還原、酸鹼、化學計量三種題型。」
- 「執行 Phase 7，先處理純文字數字與中文間距。」

## 11. 每次實作的共同安全規則

1. 不使用破壞性操作，不直接刪除舊檔。
2. 每次修改前確認哪些檔案正在被 `index.html` 與工具頁載入。
3. 每個階段完成後執行語法檢查與自動測試。
4. 涉及排版時，以固定案例進行視覺驗收。
5. 新功能先採「可關閉／可回退」設計，再逐步取代舊流程。
6. 若需額外 AI 修正，只傳送必要片段，避免整篇重新生成與浪費 Token。
