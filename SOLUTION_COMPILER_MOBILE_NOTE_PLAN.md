# 詳解編譯器、NOTE 與手機排版：近期實作計畫

## 目標與範圍

把目前「Gemini 回傳一大段文字，再由前端盡量修復」改成：

```text
題目／圖片
  -> POST /api/solve
  -> AI 僅回傳 SolutionDocument JSON
  -> schema、化學、數值與 NOTE 規則驗證
  -> 詳解編譯器產生固定渲染資料
  -> 前端元件顯示
```

本計畫只納入近期、直接影響使用者的內容：

1. AI 詳解不可直接當成 HTML／LaTeX 原文顯示。
2. NOTE 必須恢復，且觀念／選擇題也要有可點擊的說明。
3. 化學計量與平衡題必須用可驗證的三步／四步反應表呈現。
4. 手機版選項 `(A)`～`(E)` 的標籤與換行內文必須穩定對齊。
5. 首頁移除「教師工具」與「排版」兩個一般使用者入口。

以下項目不納入本期：自動配平、RDKit／OpenChemLib、Nerdamer、滴定圖與其他圖表、題庫大改、教師後台重設計。

---

## 現況與已知問題

### 1. 詳解資料流

- `js/api.js` 目前由瀏覽器直接呼叫 Gemini `generateContent`；沒有後端 API 邊界。
- `js/app.js` 取得文字後進行多個提示詞修正，再交給 `renderAiInto()`。
- `js/render.js`、`js/solution-format.js`、KaTeX 與既有反應表已能作為未來渲染器的基礎，但目前輸入仍是 AI 原文。
- 專案已有 BoardDoc 嘗試與格式檢查，然而目前主流程要求傳統 LaTeX，並禁止 BoardDoc／JSON，兩套方向彼此衝突。

### 2. NOTE 消失的可重現原因

截圖屬於觀念型選擇題，沒有多行算式。現行 `js/math-note/note-check.js` 在算式行少於 3 行時，會回傳「略過 NOTE 密度檢查」，因此 0 個 NOTE 仍視為合格。

此外，`js/math-note/note-ensure.js` 已有補 NOTE 的函式，但目前主解題流程未以它作為唯一的強制 gate。結果是：畫面仍寫「點擊數字可以觀看說明」，但詳解完全沒有可點擊 NOTE。

### 3. 手機選項對齊

- 現有選項 DOM 已拆成 `.choice-label` 與 `.choice-body`，樣式採 flex。
- 手機截圖顯示每個選項的標籤與內文仍有明顯縮排感，且長句換行時需以實機寬度驗證；不能只以桌機正常作為完成判定。
- 後續應以固定欄位寬度的 grid 取代依內容伸縮的 flex，讓每個選項的標籤欄與內文欄位一致。

## 設計原則

1. AI 只提供內容資料，不提供 HTML、CSS、`innerHTML` 片段、NOTE LaTeX 指令或版面決策。
2. 後端輸出的是 `SolutionDocument`（已驗證的渲染資料），不是直接可顯示的 HTML；前端仍用固定元件處理響應式排版。
3. 驗證失敗時，不顯示 AI 原文作為詳解；後端可要求 AI 僅修正 JSON 一次，仍失敗則回傳可理解的錯誤狀態。
4. NOTE 分為「數學／數值 NOTE」與「觀念 NOTE」，兩者都必須由編譯器建立可點擊元件。
5. 手機版與桌機版使用同一份語意 DOM；只由 CSS 調整欄寬、字級與間距。
6. 既有頁面可暫時保留供內部測試，但不再作為首頁使用者流程。

## 第一階段：建立詳解資料契約與後端邊界

### 交付物

- `SolutionDocument` JSON Schema。
- 前端統一只呼叫 `POST /api/solve`，不再由瀏覽器直接持有 Gemini API Key。
- 後端呼叫 Gemini Structured Output，回傳 JSON，不回傳 Markdown／HTML。
- Ajv 驗證 schema；不合法資料不可進入渲染器。

### 本期支援的 block

只保留目前最需要的 block，避免一開始做成過大的通用編輯器：

```text
section             段落標題
paragraph           一般說明，內容只含文字與 inline_note
choice_analysis     (A)～(E) 的逐項分析
math                已驗證的純數學式
chemical_equation   化學式／反應式資料
stoichiometry_table 三步／四步化學計量或平衡表
calculation         可重算的計算步驟
answer              最終答案與單位
```

### 三步／四步反應表（本期核心）

本期的 `stoichiometry_table` 必須明確支援兩種模式，而不是讓 AI 在段落內自由手寫表格：

| 模式 | 固定列順序 | 適用情況 |
| --- | --- | --- |
| `initial_change_final` | 起始 → 變化 → 結果（或平衡） | 一般反應計量、平衡變化可直接由 `x` 表示時。 |
| `initial_complete_reverse_equilibrium` | 起始 → 完全向右 → 再向左 → 平衡 | 平衡常數極大／極小、先假設反應近乎完全再回推時。 |

每個表格必須帶有：反應式、物種順序、係數、量的種類（莫耳數／濃度／分壓／氣體體積）、單位與各列數值。前端只依資料產生既有反應表元件，不再解析 AI 手寫的 array 或 Markdown 表格。

### 表格驗證規則

1. 反應式的物種、係數與表格欄位數必須完全一致。
2. 三步表必須符合「結果 = 起始 + 變化」；變化量必須與化學係數成比例。
3. 四步表必須符合「完全向右 = 起始 + 第一段變化」、「平衡 = 完全向右 + 再向左」；再向左的正負方向需與逆反應一致。
4. 不得省略「起始」列，也不得把最終值誤寫成變化量。
5. 非負量、限量試劑與單位一致性必須通過本機檢查；若只可做部分檢查，要向使用者明示未驗證的部分。
6. AI 可以依題意選擇模式，但模式與列資料須通過上述規則；不通過則只要求修正 table block，不能重寫整篇詳解。

### 驗證規則

1. schema：未知 block、缺欄位、型別錯誤、空答案一律拒絕。
2. 化學：已支援範圍內的反應式通過元素守恆檢查。
3. 計算：關鍵數值由計算器重算；輸出值、單位與答案須一致。
4. 排版：每一個 math／chemical block 必須通過 KaTeX（含 mhchem）預檢。
5. 安全：後端與前端均不接受 AI 指定 HTML、class、style、URL 或任意 LaTeX trust 指令。

### 尚需決定

實作前需選定 `/api/solve` 的部署位置（例如現有網站可用的 serverless／Node 服務）。此選擇不改變資料契約；未選定前可先以本機 mock API 與 fixture 完成編譯器測試。

## 第二階段：NOTE 恢復並改為編譯器職責

### 資料格式

AI 不再輸出 `\\htmlData{note=...}{...}`。資料契約改為明確的 inline note：

```json
{
  "type": "inline_note",
  "text": "銀鏡反應",
  "note": "可用來辨識具有還原性的醛類或葡萄糖。",
  "kind": "concept"
}
```

數值／算式 NOTE 則以 `kind: "quantity" | "conversion" | "factor"` 標記。詳解編譯器依 kind 產生既有 NOTE popover 所需的安全 DOM；不得把 AI 給的指令直接塞入 KaTeX。

### 覆蓋規則

| 題型 | NOTE 最低要求 |
| --- | --- |
| 觀念／選擇題 | 至少標示關鍵鑑別依據、反應性或判斷轉折；不以算式行數作為略過條件。 |
| 計算題 | 題目給定量、第一個中間量、單位換算因子、影響答案的比例／分子分母須有語意 NOTE。 |
| 反應計量／平衡題 | 除計算 NOTE 外，必須標示表格中的起始條件、變化意義與近似理由；三步／四步每一列皆須可說明。 |

### NOTE 強化規格

NOTE 的目的不是把句子切碎，也不是只替數字換一個名稱；每一個 NOTE 都要回答「這個詞／數值在此步驟為什麼重要」。

#### NOTE 類型與必標位置

| 類型 | 應標的位置 | NOTE 應說明的內容 |
| --- | --- | --- |
| `condition` | 題目給定的濃度、體積、溫壓、反應條件、物態 | 此條件限制或決定後續哪一步。 |
| `concept` | 鑑別性質、反應性、定律、近似條件 | 為何可據此判斷選項正確／錯誤。 |
| `quantity` | 第一次代入的量、關鍵中間量、係數比 | 此數字來自題目、換算或哪一個化學關係。 |
| `conversion` | mL→L、g→mol、濃度→莫耳數等換算因子 | 轉換方向與原因，不只重複單位名稱。 |
| `factor` | 分子、分母、乘積中語意不同的因子 | 係數、體積、比例各自代表什麼。 |
| `table_state` | 三步／四步反應表的列或關鍵格 | 此列是起始、完全反應後、逆向修正或真正平衡值。 |

#### 品質與密度規則

1. 每個主要推理段至少有一個 NOTE；計算段、反應表與選項判斷各自獨立計算，不可只在最後答案集中標註。
2. 觀念選擇題至少要覆蓋兩個不同的判斷依據；若逐項分析 A～E，應標記足以支持各個「正確／錯誤」結論的關鍵理由，而不是只標選項字母。
3. 計算題的每一個「新來源」數值都要有 NOTE；同一數值重複出現時可不重複標，避免滿版標籤。
4. 反應表至少標示反應式、起始列、每一段變化／回推的意義，以及近似成立的理由；不是每一格都強制塞 NOTE。
5. 禁止空泛 note，例如「數值」、「濃度」、「結果」、「公式」、「計算」；必須包含來源或化學意義。
6. 禁止只包單位、標點、選項字母、重複文字，或將太長一整句包成一個 NOTE。
7. 同一行的 NOTE 數量設上限；超出時改選最能支撐推理的錨點，保持閱讀節奏。

#### 編譯與介面行為

1. NOTE 錨點可為文字或數學內容；觀念題不能只支援「點擊數字」。
2. 編譯器在輸出前檢查：錨點非空、note 非空、類型合法、同一錨點不重複、沒有不允許的巢狀 NOTE。
3. 每個 NOTE 顯示一致的可互動提示（底線／淡色背景／圖示擇一），但不得破壞公式與選項的對齊。
4. 手機上採點擊開啟、再次點擊或點擊外部關閉；popover 不可超出視窗、不可遮住被點擊的關鍵內容。
5. 頂端提示依實際 NOTE 數量與類型動態顯示；沒有 NOTE 時不顯示「點擊數字可以觀看說明」。

#### NOTE 驗證器

NOTE gate 除了計數外，必須輸出可讀的診斷：

```text
coverage：缺少哪一段／哪一類 NOTE
quality：哪一個 note 太泛、重複或沒有說明理由
anchor：哪一個錨點不適合點擊或破壞排版
ui：手機 popover 是否超出可視範圍
```

修正時只傳回缺漏的 note block 與診斷碼，要求 AI 補齊資料；不能再以「重寫整篇詳解」處理 NOTE 問題。

### 修正現行策略

1. 移除「少於 3 算式行就略過 NOTE」作為通過條件。
2. 合併 `NoteCheck`、`NoteEnsure` 與 `ensureQualityReply()` 的職責，形成單一 NOTE gate。
3. NOTE 不足時，只請 AI 修正 `SolutionDocument` 的 note 欄位；不可重寫整篇詳解、改變選項判斷或答案。
4. 頂端提示文字改為符合實際互動的描述，例如「點擊有底線的關鍵詞或數值可看說明」；沒有 NOTE 時不得顯示此提示。
5. 為截圖所示的觀念選擇題建立 fixture，驗證至少有數個可點擊概念 NOTE。
6. 建立 NOTE 品質 fixture：空泛 note、只包單位、重複 note、過長錨點、觀念題 NOTE 過少、計算題 NOTE 集中在答案等案例都必須被拒絕。

## 第三階段：手機選項對齊

### DOM 與 CSS 方針

1. 保留 `.choice-option > .choice-label + .choice-body` 的語意結構。
2. 將選項容器改為兩欄 CSS grid：固定、可預測的標籤欄 + `minmax(0, 1fr)` 的內文欄。
3. `(A)`～`(E)`、較寬的標籤與多行選項均使用同一條對齊基準；內文換行必須回到內文欄起點，不可回到選項最左側。
4. 不用負 margin、絕對定位或依特定題目字數調整。

### 必測視窗

- 320 px、375 px、390 px、430 px 寬度。
- iPhone 安全區域（safe-area）與 Android Chrome。
- 桌機 768 px 以上回歸測試。
- 單行選項、兩行選項、含 KaTeX、含 NOTE、含多段選項，以及 `(A)`～`(E)` 全組。

### 完成條件

- 同一組選項的標籤左緣一致、內文左緣一致。
- 長選項的第二行起不與標籤重疊、不額外縮排。
- 不出現水平捲動、文字裁切或不同選項欄位跳動。

1. 從 `index.html` 首頁 header 移除「教師工具」與「排版」連結。
2. `teacher-tools.html`、`solution-format.html` 先保留作為內部／管理用測試頁，不在一般學生流程曝光，也不刪除既有模組。

   - AI 正在理解題目
   - 正在驗證化學與數值

4. 前端只接收編譯後 `SolutionDocument`；不再將 Gemini 原始文字直接交給 `innerHTML`。

## 測試與驗收

### 必要 fixture

1. 純觀念選擇題：至少有概念 NOTE，且選項 A～E 手機對齊。
2. 莫耳濃度／稀釋題：數字、換算因子與單位 NOTE 正確，數值可重算。
3. 三步反應表：起始 → 變化 → 結果（或平衡）的欄位、係數、列運算與答案一致。
4. 四步平衡表：起始 → 完全向右 → 再向左 → 平衡的列順序、方向、列運算與近似說明一致。
5. NOTE 品質：觀念、計算與反應表三類內容皆有足夠且具體的 NOTE；不可出現空泛、重複、只包單位或過長錨點。
6. NOTE 手機互動：點擊文字／數值錨點後 popover 位於可視範圍，且不破壞選項與公式排版。
7. 無效 LaTeX、無效 JSON、未知 block：不得顯示原始 AI 內容。
8. NOTE 缺失：觸發只補 note 欄位的修正，不可改答案。

### 驗收標準

- 任一詳解在顯示前都通過 schema 與渲染預檢。
- 觀念題不再因「算式不足」而跳過 NOTE。
- NOTE 不只達到數量門檻，還須通過覆蓋、內容品質、錨點位置與手機介面檢查。
- 化學計量與平衡題只要需要物種量追蹤，就必須使用已驗證的三步或四步表；不可退回段落內手寫表格。
- 手機 320～430 px 選項排列與桌機相同語意且對齊。
- 首頁不再顯示教師工具與排版入口。
- 現有化學結構、反應表與 KaTeX 詳解不回歸。

## 實作順序

1. 先補 fixture 與手機視窗測試，鎖定 NOTE／選項問題的重現方式。
2. 定義 `SolutionDocument` schema 與既有 renderer 的 block 對照。
3. 建立 `/api/solve` 邊界與 Structured Output + Ajv 驗證。
4. 實作 NOTE 編譯器與單一 NOTE gate。
5. 改選項 grid 排版並完成跨裝置視覺驗證。
6. 最後收斂首頁入口與載入狀態。

每一階段都須先通過既有 JavaScript／渲染測試，再進下一階段；不在本期加入遠期工具或大規模重構。
