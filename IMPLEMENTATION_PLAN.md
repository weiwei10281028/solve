# 高中化學 AI 解題知識庫與驗證工具：實作計畫

## 專案目標

將既有的題目與教師詳解 Markdown，轉成可檢索、可驗證、可持續修正的「教師解題知識庫」。

AI 的職責是理解題意與依教師風格講解；程式的職責是保存結構化資料、限制不合理套用、檢查答案與化學計算的一致性。

### 必須達成的原則

1. 不可因為題目表面相似就直接套用舊詳解。
2. 解題模板是檢查清單與方法地圖，不是固定劇本。
3. 指定答案是強驗證條件：逐項證明／反證，不可硬湊理由。
4. 化學反應式、莫耳比、單位與計量表應盡量由本機規則檢查。
5. Markdown 詳解只在首次整理或信心低時呼叫 AI；整理結果必須永久儲存，避免反覆耗用 Token。
6. 所有新工具優先採用現有純前端架構與 JSON／Markdown 資料，不急著導入後端或外掛。

## 現況限制

- 專案已有 `db-import.html`、`database/`、`js/database.js`、`js/db-rules.js` 與化學結構／格式相關程式。
- 現有資料庫解題功能不可直接重新啟用；必須先改造成「方法參考與風格參考」，而不是相似題答案套用。
- 保留既有 UI、題目解題流程、格式處理與使用者資料；不要進行大範圍重寫。

## 開發規則

- 每一階段開始前，先檢視相關現有檔案與資料格式；避免讀取無關檔案。
- 先完成可用的最小版本，再擴充功能。
- 每個頁面均需可在本機靜態網站執行，並提供清楚的操作說明。
- AI 回傳的分析資料必須要求 JSON 格式，並在前端檢查欄位完整性。
- 判斷、計算、字串比對等可由程式完成的工作，不呼叫 AI。
- 不要新增需要付費、雲端資料庫或使用者先安裝外掛的必要依賴；若新增選用依賴，先說明用途與替代方案。

## Phase 0：盤點與資料標準

### 目的

定義教師詳解、方法與驗證結果的共同資料格式，作為所有後續網頁的資料契約。

### 工作

1. 檢查目前 `database/` 的 Markdown front matter、`methods.json`、匯入器與解析器。
2. 設計標準題目資料格式（可先以 JSON 為主，Markdown 保留原文）。
3. 設計欄位：
   - `id`、`subject`、`grade`、`chapter`、`concepts`
   - `question_type`、`question_text`、`options`、`teacher_answer`、`solution_text`
   - `known_conditions`、`target`、`constraints`
   - `method_ids`、`method_summary`、`common_mistakes`
   - `style_tags`、`equations`、`tables`
   - `review_status`、`confidence`、`updated_at`
4. 設計方法資料格式：
   - 名稱、適用條件、禁用條件、必要檢查、候選步驟、易錯點、代表詳解 ID。
5. 建立 3～5 筆範例資料與格式文件。

### 完成條件

- 有清楚的資料格式文件。
- 現有 Markdown 至少可轉成一筆標準資料。
- 資料能保留原始詳解，且可補充人工校正欄位。

## Phase 1：知識庫整理台（第一個正式網頁）

### 新頁面

`knowledge-studio.html`

### 目的

將教師的 Markdown 題目／詳解匯入、分析、校正，轉成正式知識庫資料。

### 功能

1. 拖曳或選擇多個 `.md` 檔案。
2. 本機解析題目、答案、詳解、YAML front matter、表格與 LaTex。
3. 顯示待補欄位：章節、考點、題型、方法、易錯點、風格標籤。
4. 提供「AI 建議分類」按鈕；只將單篇或使用者選取的資料送往 API。
5. AI 回傳固定 JSON，例如：

```json
{
  "concepts": ["限量試劑"],
  "question_type": "反應計量",
  "known_conditions": ["反應完全", "兩反應物的質量已知"],
  "target": "產物質量",
  "method_ids": ["reaction-stoichiometry", "limiting-reagent"],
  "common_mistakes": ["未比較 n/係數"],
  "style_tags": ["先寫反應式", "逐步列式", "答案附單位"],
  "confidence": 0.82
}
```

6. 使用者可直接修改後按「確認收錄」。
7. 將確認資料儲存為 JSON 索引與原始 Markdown，不重複呼叫 AI。
8. 顯示「未分類／待確認／已收錄」狀態與搜尋篩選。

### 不做的事

- 不在此階段以相似度直接替新題產生答案。
- 不自動相信 AI 分類結果；必須保留人工校正。

### 完成條件

- 可匯入 10 篇 Markdown。
- 可手動校正並保存結構化欄位。
- 可針對單篇呼叫 AI 建議分類並保存結果。

## Phase 2：解題方法庫

### 新頁面

`method-library.html`

### 目的

將「教師怎麼想」整理成可彈性使用的方法地圖，不把它變成僵化模板。

### 功能

1. 顯示、建立、修改、合併方法卡。
2. 每張方法卡需含：
   - 適用條件
   - 禁用／不適用條件
   - 必須檢查事項
   - 可選步驟（不是固定步驟）
   - 常見錯誤
   - 關聯代表詳解
3. 新題使用方法前，必須先檢查適用與禁用條件。
4. 支援同題選出多個候選方法並標示理由。

### 範例

`限量試劑法`：

- 適用：反應式已知，至少兩種反應物有定量資料，要求產物或剩餘量。
- 必查：反應是否完全、係數是否配平、各物質是否已統一為莫耳數。
- 不適用：只給單一反應物且不涉及剩餘量時，不應強制比較限量試劑。

### 完成條件

- 能建立與編輯方法卡。
- 能從知識庫條目連結到相關方法。

## Phase 3：題目分析器

### 新頁面／整合功能

`problem-analyzer.html`，之後再整合回主解題頁。

### 目的

在寫詳解前先分析新題，避免以關鍵字或相似題直接決定方法。

### 流程

1. 本機抽取明顯線索：選項、數字、單位、反應箭頭、表格、圖表標題。
2. 從已收錄的結構化資料篩選同章節／同題型／同必要條件的候選方法。
3. AI 僅針對題目與少量候選方法，輸出結構化判題結果。
4. 顯示：
   - 考點與題型
   - 已知條件與求解目標
   - 候選方法及適用理由
   - 不適用方法及排除理由
   - 必須驗證的項目
5. 僅在確認題型與方法後，才進入詳解生成。

### 完成條件

- 題目分析結果可供主頁解題流程使用。
- 不會直接把某一篇相似詳解當作答案來源。

## Phase 4：指定答案驗證器

### 新頁面／整合功能

`answer-audit.html`

### 目的

當教師提供答案（例如 A、B）時，要求系統逐項驗證；不能用提示詞硬將原本的判斷改寫成指定答案。

### 流程

1. 將教師答案解析為選項集合或數值答案。
2. AI 產出每一選項的「題目依據、計算／推理、成立或不成立原因」。
3. 程式檢查：
   - 正確選項集合是否剛好等於教師答案。
   - 被標正確的選項是否有支持理由。
   - 被標錯誤的選項是否有明確反證。
4. 若初步判斷與教師答案衝突，呼叫一次「針對衝突選項重檢」；不可直接要求 AI 硬改。
5. 若重檢後仍衝突，在教師介面顯示警示：可能是 OCR、題幹、圖表、答案或題意歧義問題。
6. 學生版只有在驗證通過後才輸出正式詳解。

### 禁止行為

- 不得以「依參考答案選」作為理由。
- 不得把實際成立的選項硬說錯。
- 不得將舊題條件帶入新題。

### 完成條件

- 對多選題能檢查各選項標記與指定答案的一致性。
- 衝突時能顯示教師端警示，而非產生偽造詳解。

## Phase 5：化學計量與反應表檢查器

### 新頁面／整合功能

`chemistry-workbench.html`

### 目的

將化學方程式、莫耳比、單位與反應計量表從純文字變為可檢查結構。

### 功能

1. 輸入／解析反應式，保存反應物、生成物、係數、狀態。
2. 檢查方程式係數與元素守恆；無法確定時標為需人工確認。
3. 建立反應計量表：起始量、變化量、反應後量、單位、莫耳數。
4. 計算並檢查：
   - 莫耳數換算
   - 莫耳比
   - 限量試劑（比較 n/係數）
   - 理論產量、實際產率、剩餘量
   - 單位一致性
5. 讓 AI 只負責將已驗證的計算過程寫成教師風格詳解。

### 完成條件

- 可用一筆已知反應式與數據，產生並驗證計量表。
- 明顯的係數、比例與單位錯誤會被指出。

## Phase 6：教師測試與回饋台

### 新頁面

`evaluation-lab.html`

### 目的

建立回歸測試集，讓每次調整規則或提示詞後都能量化是否進步。

### 功能

1. 將題目標記為：標準題、易混淆題、指定答案題、表格題、方程式題。
2. 記錄預期題型、方法、答案與品質規則。
3. 批次執行本機檢查，並可選擇性執行 API 測試。
4. 教師快速標記：
   - 題型誤判
   - 方法誤用
   - 答案錯誤
   - 答案正確但講解不像教師
   - 表格／方程式／單位錯誤
5. 彙整最常見的錯誤，回連到方法卡或資料分類待修項。

### 完成條件

- 能保存至少 20 題測試案例。
- 可看到每種錯誤的數量與案例。

## API Token 控制策略

### 不使用 API 的工作

- Markdown／JSON 解析、保存與搜尋。
- YAML 欄位檢查。
- 指定答案與最終答案的字串／集合比對。
- 選項正誤標記的一致性檢查。
- 方程式元素計數、係數驗證、莫耳比、單位與算式檢查。
- 測試集統計與篩選。

### 只在必要時使用 API

- 首次為尚未分類的詳解提出標籤與方法建議。
- 新題的語意判題與候選方法選擇。
- 正式教師風格詳解生成。
- 題目與指定答案衝突時的針對性重檢。

### 節流規則

- 分類結果一經人工確認即保存，不重複分析相同 MD。
- 先用本機規則縮小候選資料，再將少量內容送給 AI。
- 不因答案格式小錯就整篇重寫；先採本機修正或針對單一問題重檢。
- 每次 API 請求記錄用途、輸入大小、結果與是否採用，供日後優化。

## 建議啟動方式

先執行 Phase 0 與 Phase 1，只使用 10～20 篇最具代表性的教師詳解驗證格式。確認分類、方法卡與風格標籤確實符合教師需求後，再做題目分析與答案驗證。

不要先重新啟用舊的相似題直接配對機制。
